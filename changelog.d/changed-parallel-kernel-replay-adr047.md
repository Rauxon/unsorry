Gate A's kernel replay stays **serial by default** with a raised timeout, instead of auto-parallelising by detected RAM (ADR-047/SPEC-047-A). A FULL replay (forced by any gate/infra change, ADR-033) was crossing the 60-min `gate-a-replay` timeout on the growing active library; the durable fix on memory-bound runners is the **60 → 120 min** timeout (one `leanchecker` already needs ~7–10 GB, so parallelism can't help a 16 GB runner). An earlier attempt to auto-size concurrency from `/proc/meminfo` was unsafe in containers — it reports the *host's* memory, not the cgroup limit, so it ran 7 concurrent `leancheckers` and OOM-killed every chunk (exit 137). Replay therefore now runs serial unless an operator explicitly opts in via **`UNSORRY_REPLAY_JOBS`** (a runner with known RAM). Soundness unchanged: concurrency only affects wall-clock, never per-module kernel verdicts. The complementary lever is ADR-041 archiving (keep the active set small so the serial full replay stays short).
