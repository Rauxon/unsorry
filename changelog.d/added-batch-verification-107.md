**Batch verification — K proofs per Gate A run (ADR-107 / SPEC-107-A, D1b).** The queue dispatcher can now combine up to `UNSORRY_BATCH_SIZE` independent queued `queued/prove/*` branches into ONE `batch/prove/<hash>` PR, so Gate A pays its dominant ~12–20 GB mathlib env-load *once* for K proofs instead of once per proof — the lever that drains the homogeneous template backlog several× faster on a fixed verifier budget (#6751 §5). Off by default (`UNSORRY_BATCH_SIZE=1` = today's one-PR-per-proof). Selection picks pairwise file-disjoint branches in dispatch order (ADR-075/106); a batch is one governor admission (ADR-058); it is enrolled with a **merge commit** (not squash) so each constituent `prove(<goal>)` stays in history and the leaderboard resolves every batched proof's merge-time + author with no leaderboard change. Soundness is untouched (ADR-049, p=1): Gate A re-verifies every proof from scratch and no proof trusts another. A genuine batch failure redispatches its constituents singly via the new `batch-recovery-janitor` (a cancellation cascade is re-run, mirroring ADR-105), so the K−1 good proofs never strand. Pure selection/dedup/recovery logic in the unit-tested `tools/dispatch/batch.py`. Refs #5683, #5678.
