Fixed the scheduled `queue-dispatcher` workflow opening only **one** PR per run instead of up to `UNSORRY_DISPATCH_LIMIT`. The workflow set `UNSORRY_DISPATCH_LIMIT=10` (and exposed it as a `workflow_dispatch` input) but invoked `agent.sh --dispatch-queue --once`; `--once` caps a dispatch pass to a single PR (documented behavior), so the limit was dead and the queued-proof backlog drained at ~1 PR per scheduled tick. The workflow now runs one pass without `--once` and sets `UNSORRY_GOVERNOR_WAIT=0` (single pass then exit), so each scheduled run dispatches up to the configured limit, still bounded by the ADR-058 governor (open-PR and in-flight Gate A caps). agent.sh behaviour is unchanged.
