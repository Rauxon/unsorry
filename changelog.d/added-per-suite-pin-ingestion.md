`import_benchmark --build` now elaborates each benchmark suite **in a suite-scoped lake project pinned to the suite's own `(toolchain, mathlib rev)`** instead of the single repo-wide pin (ADR-099 / SPEC-099-A §1, #6381). A new `tools/intake/verifier_context.py` scaffolds `targets/<suite>/_verify/{lean-toolchain, lakefile.toml, lake-manifest.json}` (reusing the `tools/archive/apply.py` template) and warms its mathlib cache; `--build` now requires `--manifest` (the suite's native `lake-manifest.json`, operator-supplied) and a `--no-warm-cache` escape hatch for offline re-runs. A **pin guard** aborts if the verifier-context rev differs from the recorded `--mathlib`, so a suite's `.aisp` pin can never diverge from the context that classified it. `--build` is also hardened to run a **real `lake env lean` of the actual statement** (not just the `foralltype` battery proxy), closing the probe-vs-build gap that passed 4 non-building goals in #6371; a genuine build failure quarantines with the distinct reason "does not build under the suite pin". The same repo-pin bug in `skeleton_validate --build` (`_check_build`) is fixed to use the suite `_verify` context when present. For native-pin suites this drops drift quarantines (e.g. `putnam-1966-b5`'s removed `Finset.toSet`) toward ~0. Hermetic tests inject the `lake` subprocess seam — no Lean needed.
